# Issue 001: 有碎片时提示"今天还没有记录碎片"，生成日记条件过于苛刻

## 描述
用户反馈：有时明明已经添加了碎片，点击"生成今日日记"却提示"今天还没有记录碎片"。生成日记的入口限制过于严格，应当在有内容时即可生成。

## 根因分析

`DiaryViewModel.generateDiary()` 当前逻辑存在两个问题：

### 1. 时区不一致导致"今日"匹配失败
- 代码侧使用 `LocalDate.now()`（系统本地时区）获取当前日期字符串
- Room DAO 使用 `date(createdAt/1000, 'unixepoch')`（UTC 时区）转换时间戳
- 在 UTC+8 时区的凌晨 0:00–8:00，`LocalDate.now()` 已经是"今天"，但 UTC `date()` 返回的还是"昨天"
- 导致用户凌晨记录的碎片无法被识别为"今日碎片"

### 2. 缺少回退机制，且状态过滤过于严格
- 只查询当日碎片，且过滤掉 `PipelineStep.COMPLETED`
- 若当日无未完成碎片，直接报错，不会尝试使用其他日期的碎片
- 用户期望：当日有内容 → 用当日的；当日没有 → 用当前所有未完成的碎片生成

## 影响范围
- `DiaryViewModel.generateDiary()`
- `FragmentDao.getFragmentsByDate()`
- `DiaryRepository.getFragmentsByDate()`

## 修复方案

1. **修复时区**：DAO 查询改为按本地时区的时间戳范围（`startOfDay ..< endOfDay`）查询，而非 SQLite UTC `date()` 函数
2. **放宽条件**：当日未完成碎片为空时，回退到所有未完成碎片（不限日期）
3. **放宽状态**：`COMPLETED` 状态的碎片仍过滤（已完成的不需要再生成），但回退逻辑确保用户随时可以生成

## 验收标准
- [x] 凌晨添加的碎片能被正确识别为"今日碎片"
- [x] 当日无碎片时，如果有其他日期的未完成碎片，允许生成日记
- [x] 所有现有单元测试通过
- [x] 模拟器上手动验证通过
